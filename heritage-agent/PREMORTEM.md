# Heritage Agent Premortem
> "6개월 후, 이 프로젝트가 망했다고 가정하자. 왜 망했을까?"

---

## Premortem이란?

프로젝트 시작 전에 "이미 실패했다"고 가정하고, 실패 원인을 미리 상상하여 사전에 대비하는 분석 방법이다. 낙관 편향을 깨고, 현실적인 리스크를 직시하기 위해 수행한다.

---

## 실패 시나리오 1: 데이터가 빈약하다

### 상상된 실패

> "앱은 만들었는데, AI가 하는 말이 위키피디아 수준이었다. 발굴 보고서를 구하려고 했지만 접근이 어려웠고, 결국 인터넷 검색으로 긁어온 일반 정보만 넣었다. 시연할 때 교수님이 물었다. '이거 ChatGPT한테 직접 물어보는 거랑 뭐가 달라요?' 대답하지 못했다."

### 왜 이렇게 될 수 있나

- 발굴 보고서가 어디 있는지 모른다
- 보고서가 있어도 PDF가 스캔 이미지라 텍스트 추출이 안 된다
- 홍릉/의릉에 대한 전문 자료가 생각보다 적을 수 있다
- 데이터 수집을 미루다가 개발에만 집중해버린다

### 사전 대응

| 대응 | 구체적 행동 | 시기 |
|------|------------|------|
| 자료 출처 다각화 | 문화재청 포털, RISS 논문, 조선왕릉관리소, 국립고궁박물관 전부 탐색 | 1주차 |
| 직접 전화 | 문화재청(042-481-4650)에 학술 목적 자료 요청 | 1주차 |
| 현장 답사 우선 | 안내판, 해설사 설명, 현장 팜플렛 전부 수집 | 2주차 |
| 데이터 품질 기준 설정 | 장소당 최소 5문장 이상의 전문 정보가 없으면 MVP 대상에서 제외 | 3주차 |
| 최악의 경우 플랜B | 홍릉/의릉 대신 자료가 풍부한 경복궁으로 대상지 변경 | 4주차 판단 |

## 실패 시나리오 2: AI가 거짓말을 한다

### 상상된 실패

> "시연 중에 AI가 '이 석탑은 신라 시대에 건립되었습니다'라고 했는데, 실제로는 조선 시대였다. 청중 중에 역사학과 교수가 바로 지적했다. 프로젝트의 핵심 가치인 '근거 있는 지적 희열'이 무너진 순간이었다."

### 왜 이렇게 될 수 있나

- LLM의 할루시네이션은 구조적 한계
- 참고 자료가 부족하면 AI가 학습 데이터에서 지어낸다
- 프롬프트에서 "모르면 모른다고 해"를 넣어도 100% 방지는 불가능
- 테스트 시간 부족으로 오답을 발견하지 못한다

### 사전 대응

| 대응 | 구체적 행동 | 시기 |
|------|------------|------|
| 프롬프트 방어 | "참고 자료에 없는 내용은 절대 지어내지 마세요" 명시 | Step 3 |
| 출처 표시 | AI 응답 하단에 "참고: {data_file}" 표시 | Step 8 |
| 팩트체크 세션 | 모든 장소에 대해 3가지 페르소나 응답을 인쇄해서 팀 전원이 검증 | 10주차 |
| 안전장치 | 매칭되는 장소가 없을 때 "이 장소에 대한 전문 자료가 아직 없습니다"로 솔직하게 표시 | Step 8 |
| 시연 리허설 | 발표 전 최소 3회 리허설, 예상 질문에 대한 AI 응답 사전 확인 | 14주차 |

## 실패 시나리오 3: 팀원이 이탈한다

### 상상된 실패

> "개발 담당 팀원이 중간에 다른 과목 과제에 치여서 손을 놨다. 남은 팀원은 코드를 이해하지 못해서 유지보수도 못 했다. 결국 반쯤 완성된 앱으로 발표했다."

### 왜 이렇게 될 수 있나

- MBA 과정 특성상 여러 과목의 과제가 동시에 몰린다
- 개발자 의존도가 높은 구조 (코딩 가능 인원 1~2명)
- 팀 번아웃, 동기 저하
- 역할 분담이 불균형하면 특정 인원에게 부담 집중

### 사전 대응

| 대응 | 구체적 행동 | 시기 |
|------|------------|------|
| 주간 모임 유지 | 6층 강의실 모임 원칙 준수 (할 일 없어도 모임) | 매주 |
| 코드 공유 | GitHub에 모든 코드 올리고, README에 실행 방법 상세 기록 | 상시 |
| 페어 코딩 | 개발자 혼자 작업하지 않고, PM이 옆에서 함께 이해하며 진행 | 4주차~ |
| Claude Code 활용 | 개발자가 빠져도 PM이 Claude Code로 최소한의 유지보수 가능하도록 | 상시 |
| 마일스톤 축소 계획 | 팀원 이탈 시 지도 앨범을 빼고 가이드 기능만으로 발표하는 플랜B 준비 | 8주차 판단 |

## 실패 시나리오 4: 현장에서 작동하지 않는다

### 상상된 실패

> "개발 내내 사무실 와이파이에서만 테스트했다. 실제 홍릉에 갔더니 모바일 데이터가 느려서 Gemini API 응답이 30초 넘게 걸렸다. 사용자는 그 사이에 앱을 닫았다. 또, 사진에 GPS가 안 찍히는 폰이 있다는 걸 현장에서 알았다."

### 왜 이렇게 될 수 있나

- Gemini API 응답 시간이 네트워크에 따라 크게 변동
- 일부 스마트폰은 카메라 GPS 기본값이 꺼져 있음
- 유적지 내 셀룰러 신호가 약할 수 있음
- 실내/나무 아래 등에서 GPS 정확도 저하

### 사전 대응

| 대응 | 구체적 행동 | 시기 |
|------|------------|------|
| 로딩 UX | "AI가 설명을 준비하고 있어요..." 로딩 애니메이션 + 예상 대기 시간 표시 | Step 8 |
| GPS 대체 | GPS 없을 때 드롭다운으로 장소 직접 선택 가능하게 | Step 8 |
| 조기 현장 테스트 | 기능이 절반만 완성되어도 현장에서 1차 테스트 | 8주차 |
| 모바일 데이터 테스트 | 사무실 와이파이를 끄고, 핸드폰 핫스팟으로 테스트 | 7주차~ |
| 응답 캐싱 | 같은 장소+같은 페르소나 요청은 이전 응답 재사용 검토 | 10주차 |

## 실패 시나리오 5: 예산 0원의 벽에 부딪힌다

### 상상된 실패

> "Gemini 무료 티어의 일일 호출 한도를 시연 리허설 중에 다 써버렸다. 정작 발표 당일에는 API가 막혀서 라이브 데모를 못 했다."

### 왜 이렇게 될 수 있나

- Gemini 무료 티어: 분당 15회, 일일 1,500회 제한
- 개발 중 반복 테스트로 할당량 소진
- 발표 당일에 여러 번 시연하면 한도 초과 가능
- 무료 Streamlit Cloud의 리소스 제한

### 사전 대응

| 대응 | 구체적 행동 | 시기 |
|------|------------|------|
| API 호출 절약 | 개발 중에는 저장된 샘플 응답으로 테스트, 실제 API는 최종 검증에만 사용 | 4주차~ |
| 응답 저장 | 한 번 받은 AI 응답은 JSON으로 저장해서 같은 요청 시 재사용 | Step 6 |
| 발표 당일 대비 | 주요 시연 시나리오 3개의 응답을 미리 캐싱 | 15주차 |
| 복수 API 키 | 팀원 각자 Gemini API 키를 발급받아 비상시 교체 가능하게 | 4주차 |
| 오프라인 폴백 | API가 완전히 막히면 저장된 응답으로 시연하는 오프라인 모드 구현 | 12주차 |

## 실패 시나리오 6: "그래서 뭐가 다른데?" 질문에 답하지 못한다

### 상상된 실패

> "발표 후 질의응답에서 '이거 네이버 스마트렌즈로도 되지 않나요?', 'ChatGPT한테 직접 사진 보여주면 되잖아요'라는 질문이 나왔다. 차별점을 명확하게 설명하지 못했다."

### 왜 이렇게 될 수 있나

- Gemini/ChatGPT에 직접 사진을 보여줘도 일반적인 설명은 가능
- 네이버/구글 렌즈도 랜드마크 인식 기능이 있음
- "발굴 보고서 기반"이라는 차별점이 데이터 없이는 말뿐

### 사전 대응

| 대응 | 구체적 행동 | 시기 |
|------|------------|------|
| 비교 시연 준비 | 같은 사진을 ChatGPT에 보여준 응답 vs 우리 앱 응답을 나란히 비교하는 슬라이드 | 14주차 |
| 차별점 3가지 암기 | ① 발굴 보고서 기반 전문 정보 ② 페르소나별 맞춤 설명 ③ 지도 앨범 여행 기록 | 발표 전 |
| 데이터 깊이 증명 | "ChatGPT는 이걸 모르지만, 우리 앱은 안다"는 구체적 사례 3개 확보 | 12주차 |

## 핵심 교훈

이 Premortem에서 반복적으로 드러나는 패턴이 있다:

**"대부분의 실패 원인은 코드가 아니라 데이터와 테스트 부족에서 온다."**

- 코드는 Claude Code가 도와줄 수 있다
- 데이터는 사람이 직접 발로 뛰어야 한다
- 테스트는 현장에서 해야만 의미가 있다

따라서 Phase 1(데이터 확보)과 Phase 3(현장 테스트)에 시간을 아끼지 않는 것이 프로젝트 성공의 열쇠다.
