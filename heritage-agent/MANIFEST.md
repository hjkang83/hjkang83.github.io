# Heritage Agent Manifest
> 우리가 왜 이걸 만드는지, 어떤 기준으로 결정하는지에 대한 약속

---

## 한 줄 정의

**유적지 현장에서, 발굴 보고서 수준의 지식을 누구나 자기 눈높이로 경험하게 만드는 AI 가이드**

---

## Core Values (핵심 가치)

### 1. 근거 있는 지적 희열 (Data Integrity)

AI가 그럴듯하게 지어낸 이야기는 독이다. 우리는 유네스코, 문화재청, 국가 발굴 보고서 등 공인된 자료만을 AI의 지식 기반으로 사용한다. 사용자가 "이거 진짜야?"라고 물었을 때, 출처를 댈 수 있어야 한다. 출처를 댈 수 없으면 "모릅니다"라고 답하는 게 낫다.

**실천 기준:**
- 모든 AI 응답은 `data/` 폴더의 참고 자료에 근거할 것
- 프롬프트에 "자료에 없는 내용을 지어내지 마세요" 명시
- 할루시네이션 발견 시 데이터 보강을 우선하고, 프롬프트 트릭으로 회피하지 않을 것

### 2. 시선은 유적에, 손은 주머니에 (Experience-Centric)

사용자가 유적지까지 가서 스마트폰 화면만 들여다보게 만들면 실패다. 우리 앱은 "찍고, 듣고, 올려다본다"는 흐름이어야 한다. 사진 한 장 찍으면 나머지는 귀로 듣는다. 화면을 계속 쳐다보게 만드는 UI는 잘못된 UI다.

**실천 기준:**
- 음성 재생(TTS)을 핵심 출력으로 취급할 것
- 텍스트는 보조 수단, 음성이 주 수단
- UI 요소를 늘리고 싶은 충동이 들면 "이게 사용자의 시선을 유적에서 빼앗는가?" 자문할 것

### 3. 같은 돌, 다른 이야기 (Hyper-Personalization)

같은 석탑 앞에 서 있어도 7살 아이와 건축학과 교수의 궁금증은 완전히 다르다. 우리는 하나의 유적에 대해 여러 층위의 이야기를 제공한다. "누구에게 설명하는가"가 달라지면 톤, 깊이, 비유, 용어 전부가 달라져야 한다.

**실천 기준:**
- 최소 3가지 페르소나(어린이/일반/전문가)를 구분할 것
- 페르소나 간 응답이 "단어 몇 개만 다른 수준"이면 실패
- 같은 장소에 대해 3가지 페르소나 응답을 나란히 놓고 읽었을 때 확연히 달라야 함

### 4. 파편을 맥락으로 잇기 (Seamless Connection)

유적 현장, 박물관 전시물, 숨겨진 인근 유적은 원래 하나의 이야기다. 우리는 이 조각들을 GPS 기반으로 연결한다. "방금 보신 석탑에서 나온 유물이 여기 박물관에 있습니다"라고 말해줄 수 있어야 한다.

**실천 기준:**
- "근처 다른 볼거리" 추천 기능을 반드시 포함
- 장소 간 관계 정보를 데이터에 포함할 것
- 사용자가 "여기 말고 또 뭐 있어?"라고 물었을 때 답할 수 있을 것

### 5. 여행의 밀도를 높인다 (Enhancing Travel Density)

평범한 돌덩이를 찬란한 역사의 증거로 바꾸는 것이 우리의 일이다. 발굴 보고서 속에 잠들어 있던 해석을 현장으로 불러내서, 그냥 지나쳤을 곳에서 발걸음을 멈추게 만든다. 사용자의 탐색 비용을 줄이면서 감동의 밀도를 높인다.

**실천 기준:**
- 지도 앨범에 방문 기록이 쌓이면서 "내 여행의 궤적"이 보여야 함
- 사용자가 몰랐을 정보를 최소 1개 이상 포함하여 "아, 그랬구나" 반응을 이끌어낼 것

## Principles (운영 원칙)

### 1. 6층 빈 강의실은 우리의 아지트다 (Team Bonding First)

매주 수업 후 전원이 모인다. 할 일이 없어도 모인다. 함께 밥 먹고, 수다 떨고, 일상을 공유한다. 과제 파트너가 아니라 하나의 팀이 된다.

### 2. 결정은 민주적으로, 승복은 쿨하게 (Agile & Democratic Decision)

의견이 갈리면 충분히 토론한다. 그래도 안 되면 카카오톡 익명 투표로 정한다. 결과가 나오면 전원이 합의하고 실행한다. "내 의견이 아니었으니 대충 하겠다"는 없다.

### 3. 작게 시작하되 확실하게 (Small Start, Solid Function)

전 세계 유적지를 담으려는 욕심을 버린다. 홍릉/의릉 하나에서 완벽하게 작동하는 것이 먼저다. "좁지만 깊은 서비스"가 "넓지만 얕은 서비스"를 이긴다.

### 4. 예쁜 쓰레기보다 투박한 명품 (Function over Form)

화려한 UI에 매몰되지 않는다. 초기에는 백엔드 로직과 프롬프트 엔지니어링에 자원을 집중한다. 디자인은 기능이 안정된 후에 다듬는다. "예쁜데 정보가 틀린 앱"보다 "투박한데 답변이 정확한 앱"이 낫다.

## 이 Manifest의 사용법

- 기능을 추가할지 말지 고민될 때 → 핵심 가치에 부합하는가?
- 의견이 갈릴 때 → 원칙에 비추어 판단
- 코드 리뷰할 때 → 이 기준에 맞게 구현되었는가?
- 방향을 잃었을 때 → 맨 위 "한 줄 정의"로 돌아온다
